01

問題設定與一句話總結

在凍結的視覺語言模型上,用最小的可訓練參數量問一個工程問題:rank 跟 alpha 各值多少錢、買到多少準確率?

任務:把開源視覺語言模型 Qwen2-VL-2B-Instruct 用 LoRA 在 VQAv2(視覺問答)上做監督式微調,並系統性地掃描 LoRA 的兩個核心超參 r(rank)與 alpha,量化「容量 ↔ 準確率」的取捨曲線。

一句話結論:在這個 5000 筆子集上,rank 把參數量拉 8×(r4→r32)只換到 +1.4 個百分點的 exact-match(0.699→0.713);alpha/r = 2× 在多數 rank 上穩定優於 1×;r8_a16 是準確率/參數量 CP 值最高的工作點。容量不是這份資料的瓶頸。

為什麼這個 demo 對 SA / ML 職位有訊號
它同時展示三件事:(1) 能把一個多模態模型的訓練 pipeline 從零跑通並除錯;(2) 懂 PEFT 的數學與工程取捨,不是 call API;(3) 會設計 ablation、讀曲線、下可辯護的結論 — 這正是 Solutions Architect 幫客戶調模型時要做的事。
02

Dataset — VQAv2 內容

VQAv2 是視覺問答的標準 benchmark,設計上刻意「去語言先驗」,逼模型真的看圖。

每一筆長什麼樣

一筆 = (image, question, answers)。圖來自 COCO;問題是人寫的自然語言;答案由 10 位標註者各給一個短詞,另有一個 multiple_choice_answer(最高票答案)。

{
  "image": <PIL.Image, COCO 照片>,
  "question": "What color is the umbrella?",
  "answers": [{"answer":"red"}, {"answer":"red"}, ... ×10],
  "multiple_choice_answer": "red"      # 我們訓練/評估用的 gold
}

為什麼是 "v2"

VQA v1 有嚴重的語言偏差 — 很多問題不看圖、純猜統計先驗就能答對(「天空什麼顏色」永遠答藍)。VQAv2 為每個問題配一張答案相反的互補圖(complementary pairs),把這條捷徑堵死,所以 v2 上的分數更能反映真正的視覺理解。面試常考這點。

我的取樣與切分

  • 來源 repo:lmms-lab/VQAv2 的 validation split(parquet 原生)。原始 HuggingFaceM4/VQAv2 是 loading-script 形式,datasets≥3 已不支援,會直接報錯。
  • shuffle(seed=42) 後取 5000 筆,再 train_test_split(test_size=0.2) → 4000 train / 1000 eval。
  • 固定 seed 確保每個 sweep config 看到的是同一批 train/eval,結果才可比。
限制 / 誠實揭露
用的是 validation 切出來的子集當 train+eval,不是官方 test-std;5000 筆也遠小於完整 VQAv2(~120 萬 QA)。所以絕對分數只在這個受控設定內有意義,sweep 之間的相對趨勢才是重點。
03

Model — Qwen2-VL-2B 架構

一個能吃任意解析度圖片的視覺語言模型;我們只在它的注意力投影上掛 adapter,主幹全凍。

三大組件

  • Vision encoder(ViT):把圖切成 28×28 的 patch,每個 patch → 一個視覺 token。Qwen2-VL 的特色是 Naive Dynamic Resolution — 不強制縮成固定大小,圖越大 token 越多。
  • VL merger / projector:把視覺 token 投影到語言模型的 embedding 空間,並做空間壓縮(相鄰 patch 合併),降 token 數。
  • LLM decoder(Qwen2,~2B):把 [視覺 token + 文字 token] 當一條序列,autoregressive 生成答案。用 M-RoPE(multimodal rotary position embedding)分別編碼時間/高/寬。

載入(關鍵兩行)

model = Qwen2VLForConditionalGeneration.from_pretrained(
    "Qwen/Qwen2-VL-2B-Instruct", torch_dtype=torch.bfloat16)
model = model.to("cuda")        # 不用 device_map="auto"
工程細節:為什麼不用 device_map="auto"
device_map="auto" 在單卡上會誤判,把模型一部分 offload 到 CPU,而 transformers≥4.5x 的 Trainer 偵測到 hf_device_map 就直接拒絕訓練。2B 模型在 L4 24GB 上綽綽有餘,直接 .to("cuda") 最乾淨。這是實跑才會踩到、純看教學不會知道的坑。

Pre-processing — collator 與 token masking

監督式微調的精髓在 label masking:只在「答案」那幾個 token 上算 loss,其餘全部 -100。

1 · 組對話

user: image + "Question: …" / assistant: answer

→
2 · chat template

套 Qwen2-VL 模板,圖展開成 <image_pad> 佔位 token

→
3 · processor

文字 tokenize + 圖 patch 化 + padding,出 input_ids / pixel_values

→
4 · mask labels

pad、image、prompt 全設 -100,只留 answer

label masking 三步

labels = input_ids.clone()
labels[labels == pad_id]          = -100   # 不學 padding
labels[input_ids == image_token]  = -100   # 不學圖的佔位 token
labels[i, :prompt_len[i]]         = -100   # 不學問題,只學答案

結果:cross-entropy 只在 answer span 上回傳梯度。模型學的是「看到這張圖+這個問題,該吐什麼答案」,而不是去重建問題或圖。

最深的一個 bug:truncation 砍掉 image token
原本 collator 開了 truncation=True, max_length=512。Qwen2-VL 一張圖會展開成數百個 <image_pad> token,512 截斷把一部分砍掉 → 文字宣稱的 image-token 數和 input_ids 實際數對不上 → processor 直接 ValueError。修法:不截斷,改用 image_processor.max_pixels 限制圖的解析度來控 token 數。截斷會切爛多模態對齊,這是 VLM 微調的經典陷阱。
05

LoRA — rank / alpha 的數學意義

凍結原權重,只學一個低秩增量;rank 決定容量,alpha 決定這個增量被放多大。

核心公式

對一個原權重矩陣 W₀ ∈ ℝ^(d×k),LoRA 不動它,改學兩個小矩陣 B ∈ ℝ^(d×r)、A ∈ ℝ^(r×k),其中 r ≪ min(d,k):

h = W₀·x  +  (alpha / r) · (B·A)·x
            └── 凍結 ──┘   └── 可訓練低秩增量 ──┘
A ~ N(0, σ²)  初始化,  B = 0  →  起點 ΔW = 0,訓練從原模型出發
  • r(rank) = 低秩瓶頸的寬度,直接決定可訓練參數量:每個 target 模組 = r·(d+k) 個參數。r 越大,ΔW 能表達的方向越多 → 容量越大。
  • alpha = 縮放因子的分子。實際套用的是 alpha/r。它不增加參數,只放大/縮小這個低秩增量對輸出的貢獻 — 行為上類似給 adapter 一個額外的有效學習率。
面試關鍵:為什麼掃 alpha=r 和 alpha=2r 兩條線
因為真正影響訓練的是比值 alpha/r,不是 alpha 絕對值。固定比值、只放大 r,測的是「容量」效果;固定 r、把 alpha 從 r 調到 2r,測的是「scaling / 有效學習率」效果。兩個軸拆開掃,才能歸因。

為什麼只掛在 q_proj / v_proj

這是原始 LoRA 論文的標準目標集 — attention 裡 query 和 value 投影是 CP 值最高的注入點。trainable_params 因此極小:最大的 r32 也只佔全模型 0.197%。把 k_proj/o_proj/MLP 也加進來是最直接的下一步 ablation。

訓練參數與設定

每個參數都有理由,面試官會逐個問「為什麼這個值」。

參數值為什麼
epochs1小資料 + LoRA,1 epoch 足以看出趨勢,避免過擬合 & 省時
lr2e-4LoRA 常用區間(比 full-FT 的 ~2e-5 高一個量級,因只訓練少量參數)
schedulercosine + 5% warmupwarmup 穩住前期,cosine 平滑收斂
batch4 × 4 = 16per-device 4 受 L4 顯存限制,gradient accumulation 補到有效 16
precisionBF16L4 native 支援;比 FP16 動態範圍大,不易 overflow
dropout0.05LoRA 層的輕度正則
seed42固定資料切分,讓 8 個 config 公平可比

Sweep 矩陣

r ∈ {4, 8, 16, 32} × alpha ∈ {r, 2r} = 8 個 run。每個 ~21 分鐘(含 1000 筆序列化 eval),單卡跑完約 2h47m,跑完自動關機省成本。

Evaluation — VQA exact match

指標看似簡單,但「正規化」那一步藏了 VQA 的標準慣例。

對 1000 筆 held-out,逐筆 generate() 出答案,跟 gold 做正規化後的精確比對:

def vqa_normalize(s):
    s = s.lower()
    s = re.sub(r"[^\w\s]", " ", s)        # 去標點
    s = re.sub(r"\b(a|an|the)\b", "", s)  # 去冠詞
    return re.sub(r"\s+", " ", s).strip() # 收斂空白

去冠詞/標點是 VQA 官方 metric 的慣例("a red car" 跟 "red car" 該算對)。本 demo 用單一 gold(multiple_choice_answer)做 exact match;官方 VQA accuracy 是對 10 個標註答案取 min(matches/3, 1) 的軟分數,比這個寬鬆 — 是已知的簡化。

已知瓶頸:eval 是序列化 generate
1000 筆逐筆 generate() 沒有 batch,佔了每個 run 的大頭時間。要加速可改 batched generation 或減少 eval 筆數 — Day2 規模下還可接受,擴大 sweep 時必須先優化。

三個歸因

  1. rank 單調、但邊際遞減:r4(~0.70)→ r32(~0.713),參數量 8× 只換 +1.4 pt。圖 2 的點往右上爬但斜率漸平 → 容量不是這份資料的瓶頸。
  2. alpha/r = 2× 多數情況更好:r4(0.699→0.700)、r16(0.705→0.708)、r32(0.712→0.713)都是 2× 勝。較大的 scaling 像給 adapter 更高的有效學習率,小資料下有利。r8 是唯一反例(0.707→0.705),差距在 run-to-run 噪聲級。
  3. CP 值最高是 r8_a16:0.705 用了 r32 的 ¼ 參數。r32_a64 雖最高分,但多付 4× 參數只買 +0.8 pt。要選一個上線就選 r8。
2000 vs 5000 樣本的交叉驗證
我先跑過一輪 2000 筆快版,趨勢完全一致、分數高約 0.5–1 pt。小 eval set 讀起來偏樂觀(樣本少、方差大),5000 的數字才是可信的那組。能主動講這個對比,顯示懂統計可信度,不是只報一個數。
09

亮點與限制

面試官最愛問「這專案的限制是什麼」— 主動講比被挖出來好。

亮點

  • 端到端可重現:config 驅動、固定 seed、結果自動寫 CSV、一鍵畫圖、跑完自動關機。換一台機 git clone 就能複現。
  • 真的除過錯:SFTTrainer→Trainer、script-dataset 失效、jinja2 版本、device_map、collator 簽名、truncation 砍圖 token — 六個只有實跑才會炸的問題逐一解掉。
  • 結論可辯護:不只報分數,拆出 rank(容量)vs alpha/r(scaling)兩軸歸因,並用雙樣本量交叉驗證可信度。
  • 成本意識:單張 L4、sweep 自動關機、先小樣本驗 pipeline 再上大樣本。

限制(誠實清單)

  • 資料規模小:5000 筆子集、validation 切分,非官方 test-std。絕對分數不可外推。
  • metric 簡化:單 gold exact match,非官方 10-answer 軟分數。
  • target 模組窄:只 q/v proj,沒掃 k/o/MLP,也沒掃 dropout / lr / epochs。
  • eval 未優化:序列化 generate 是時間瓶頸。
  • 單一 seed:沒跑多 seed 估方差,~0.1–0.2 pt 的差異要當噪聲看。

下一步

多 seed 估誤差棒 → 擴 target 模組 → batched eval → 加 QLoRA(4-bit)看顯存/精度取捨 → 跟 full fine-tune 對照上界。

10

綜合面試問答

點開即答。挑高機率、會追問的那些。

LoRA 為什麼能省這麼多?它凍結了什麼?
凍結全部原始權重,只訓練每層注入的低秩矩陣 A、B。可訓練參數從 20 億掉到一兩百萬(<0.2%)。連帶好處:optimizer state(Adam 的動量/方差)也只需存這一小撮 → 顯存大降;且 adapter 可插拔,一個 base 配多個任務 adapter。
alpha 跟 rank,哪個比較重要?
看你要什麼。rank 給容量(能表達的增量方向數),alpha/r 給 scaling(增量被放多大,類似有效學習率)。本實驗 rank 提供主要增益但遞減;alpha/r=2× 是穩定的小幅加成。真正起作用的是比值 alpha/r,所以調參時固定比值掃 rank、或固定 rank 掃比值,別兩個一起亂動。
B 為什麼初始化成 0?
讓訓練起點的增量 ΔW = B·A = 0,模型一開始完全等於原始預訓練模型,不破壞已學到的能力,再從這個安全起點微調。A 用隨機高斯、B 用 0,是為了 break symmetry 同時保證零起點。
為什麼準確率只有 ~0.71,不是更高?
三個原因:(1) base 模型本來就有不錯的零樣本 VQA 能力,微調是在已高的起點上小幅推進;(2) 只訓練 q/v proj、<0.2% 參數、1 epoch、小資料;(3) 我用嚴格的單-gold exact match,比官方 10-answer 軟分數低。重點不是衝高分,是受控比較 rank/alpha。
VQAv2 跟 v1 差在哪?為什麼重要?
v1 有語言偏差,很多題不看圖靠先驗就能猜對。v2 給每題配一張答案相反的互補圖,堵掉這條捷徑,分數更能反映真實視覺理解。用 v2 = 你知道在測什麼。
多模態微調最容易踩的坑?
我實際踩到的:截斷把 image placeholder token 砍掉,導致文字宣稱的圖 token 數和 input_ids 對不上,processor 直接報錯。VLM 的圖會展開成大量 token,不能像純文字那樣隨便 truncation=max_length。正解是用 max_pixels 控解析度來限 token 數,而非截斷。
怎麼確定你的結論不是噪聲?
兩件事:(1) 固定 seed,8 個 config 看同一批資料,差異來自超參而非資料切分;(2) 另跑一輪 2000 筆快版交叉驗證,趨勢一致、分數系統性高 0.5–1 pt(小樣本偏樂觀),確認方向穩。誠實講:單 seed,~0.1–0.2 pt 的差異我當噪聲看,要更穩需多 seed 估誤差棒。
如果要上 production,你會怎麼選?
選 r8_a16:0.705 的準確率、僅 r32 的 ¼ 參數,adapter 更小、載入更快、顯存更省。最高分的 r32_a64 多付 4× 參數只多 0.8 pt,不划算 — 除非那 0.8 pt 對業務關鍵。這就是 CP 值思維。
